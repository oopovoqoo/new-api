## 第1章：注册与认证改造

### 1.1 手机号注册 + 短信验证码

**现状**
`model/user.go` 第 23-53 行，用户模型中无手机号字段。注册仅支持邮箱 + 密码或第三方 OAuth，无法满足中国大陆用户习惯及实名制要求的前置条件。

**改造需求**
- 支持手机号 + 短信验证码注册
- 支持手机号 + 短信验证码登录（无密码登录）
- 支持通过手机号找回密码
- 手机号全局唯一，不可重复注册

**方案**

后端改造：

```go
// model/user.go 新增字段
type User struct {
    // ... 现有字段 ...
    Phone         string `json:"phone" gorm:"column:phone;uniqueIndex;default:''"`
    PhoneVerified bool   `json:"phone_verified" gorm:"column:phone_verified;default:false"`
}
```

新建 `service/sms.go`，提供统一短信发送接口，底层支持切换阿里云 SMS 或腾讯云 SMS：

```go
// service/sms.go
type SMSProvider interface {
    SendVerificationCode(phone, code, ttl string) error
}

// 阿里云实现
type AliyunSMS struct { ... }
// 腾讯云实现
type TencentSMS struct { ... }
```

验证码存储（Redis，TTL 5分钟）：
- Key 格式：`sms:verify:{phone}`
- Value：6位数字验证码
- 发送频率控制：同号 1分钟限 1次，1天限 5次（见 1.2）

新增路由：
```
POST /api/sms/verification          # 发送验证码（需传 phone + 图形验证码 token）
POST /api/user/register/phone       # 手机号注册
POST /api/user/login/phone          # 手机号验证码登录
POST /api/user/password/reset/phone # 手机号找回密码
```

`controller/user.go` 扩展 Register() 函数（第 131 行起），新增 phone 注册分支。

前端改造：
- `web/src/components/auth/RegisterForm.jsx`：添加注册方式 Tab（邮箱/手机号），手机号输入框 + 60s 倒计时获取验证码按钮
- `web/src/components/auth/LoginForm.jsx`：添加手机号验证码登录 Tab

### 1.2 短信防刷：验证码方案替换

**现状**
`middleware/turnstile-check.go` 使用 Cloudflare Turnstile。Cloudflare 在中国大陆无 CDN 节点，JS 加载延迟高（通常 3-8s），用户体验差，且存在访问失败风险。

**改造需求**
替换为国内可访问的人机验证方案，防止短信轰炸。

**推荐方案：极验 GeeTest v4**

极验在国内有 CDN 覆盖，行为验证（滑动拼图）体验好，有完整的 Go SDK。

```go
// service/captcha.go
type CaptchaProvider interface {
    Verify(lotNumber, captchaOutput, passToken, genTime string) (bool, error)
}

type GeeTestV4 struct {
    CaptchaId  string
    CaptchaKey string
}
```

备选方案：腾讯防水墙（`captcha.qq.com`），同样国内CDN，适合已有腾讯云账号的团队。

短信专属限速中间件（新建 `middleware/sms-rate-limit.go`）：

```go
// Redis key: sms:rate:minute:{phone} TTL=60s
// Redis key: sms:rate:daily:{phone}  TTL=86400s
func SMSRateLimit() gin.HandlerFunc {
    return func(c *gin.Context) {
        phone := c.PostForm("phone")
        // 1分钟内同号限1次
        // 1天内同号限5次
        // 超限返回 429 Too Many Requests
    }
}
```

配置开关（`common/config.go`）：

```go
var CaptchaProvider = env.GetOrDefault("CAPTCHA_PROVIDER", "turnstile")
// 可选: "turnstile" | "geetest" | "tencent" | "none"
```

| 前端：替换 Turnstile 组件为极验 SDK | 1天 |
| **合计** | **3.5天** |

**涉及文件**

### 1.3 实名制认证（KYC）

**现状**
无任何实名认证功能。

**改造需求**
《生成式人工智能服务管理暂行办法》（2023年8月）第九条要求：服务提供者应当对用户进行真实身份核验。核验方式：姓名 + 身份证号 + 手机号三要素或二要素核验。

**方案**

数据模型（新建 `model/kyc.go`）：

```go
type UserKYC struct {
    Id           int64  `json:"id" gorm:"primaryKey"`
    UserId       int    `json:"user_id" gorm:"uniqueIndex"`
    RealName     string `json:"-" gorm:"column:real_name"` // AES 加密存储
    IdCard       string `json:"-" gorm:"column:id_card"`   // AES 加密存储
    Phone        string `json:"phone_masked"`               // 脱敏展示
    VerifyStatus int    `json:"verify_status"`
    // 0=未认证 1=认证中 2=已认证 3=认证失败
    VerifyTime   int64  `json:"verify_time"`
    FailReason   string `json:"fail_reason"`
}
```

接入第三方实名 API（推荐阿里云实人认证，次选腾讯云）：
- 请求：姓名 + 身份证 + 手机号 → 三要素核验
- 响应：一致/不一致/查询失败
- 单次调用费用：约 0.1-0.3 元，需预充值

KYC 配置开关（支持强制/可选两种模式）：

```go
// common/config.go
var KYCRequired = env.GetBool("KYC_REQUIRED", false)
// true: 未认证用户无法调用 API
// false: 鼓励认证但不强制
```

前端：`web/src/components/settings/personal/cards/AccountManagement.jsx` 新增实名认证卡片，展示认证状态、提供认证入口。

管理后台：新增用户 KYC 状态列，支持手动审核通过/拒绝（兜底人工审核）。

**工作量估算**

| 任务 | 工作量 |
|------|--------|
| 后端：model/kyc.go + controller + 路由 | 2天 |
| 后端：阿里云实人认证 SDK 对接 | 1天 |
| 前端：AccountManagement 认证卡片 | 1天 |
| 管理后台：KYC 管理页 | 1天 |
| **合计** | **5天** |

**涉及文件**
- `model/kyc.go`（新建）
- `controller/kyc.go`（新建）
- `service/kyc.go`（新建，封装第三方 API）
- `router/api-router.go`
- `web/src/components/settings/personal/cards/AccountManagement.jsx`

### 2.1 微信支付 + 支付宝（Epay 通道）

**现状**

### 2.2 原生微信支付（替代 Epay，可选）

**现状**：通过 Epay 中转，需向 Epay 支付服务费（通常 0.5-1%）。

**方案**：直接接入微信支付 v3 API，去掉中间层。

新建 `controller/topup_wechat_native.go`：

```go
// 微信支付 Native（扫码支付）
// POST /pay/transactions/native
// 响应中包含 code_url，前端生成二维码展示

// 微信支付回调（JSAPI/Native）
// POST /api/pay/wechat/notify
// 验签方式：RSA-OAEP，使用微信平台证书公钥
```

支持场景：
- Native 扫码支付（PC 端主要方式）
- JSAPI 公众号支付（微信内 H5，见 2.4）

优势：无中间商费率，资金直接到商户结算账户，T+1 结算。

**工作量**：后端 4天（含证书管理、回调验签、对账逻辑）。

### 2.3 退款系统

**现状**
代码层只有 Quota 的内部退还逻辑（`service/billing.go`），无实际资金退款功能。

**改造需求**
用户充值后若对服务不满，可申请退款，资金原路退回。符合《消费者权益保护法》相关要求。

**方案**

数据模型（新建 `model/refund.go`）：

```go
type Refund struct {
    Id          int64   `json:"id" gorm:"primaryKey"`
    UserId      int     `json:"user_id" gorm:"index"`
    TopUpId     int64   `json:"top_up_id"` // 关联充值记录
    Amount      float64 `json:"amount"`    // 退款金额（元）
    QuotaDeduct int64   `json:"quota_deduct"` // 同步扣减的 Quota
    Reason      string  `json:"reason"`
    Status      int     `json:"status"` // 0=待审核 1=已通过 2=已拒绝 3=退款中 4=已完成
    AdminNote   string  `json:"admin_note"`
    ApplyTime   int64   `json:"apply_time"`
    ProcessTime int64   `json:"process_time"`
}
```

退款条件（可配置）：
- 充值后 N 天内可申请（默认7天）
- 已消耗的 Quota 部分不退（按比例）
- 单用户每月最多申请 X 次

退款资金流向：
- Epay：调用 Epay 退款接口
- 微信支付：POST `/pay/refunds`（需约 1-3 个工作日到账）
- 支付宝：调用 `alipay.trade.refund`

前端：`web/src/components/topup/TopupHistoryModal.jsx` 在满足退款条件的充值记录行添加"申请退款"按钮，展示退款状态。

管理后台：退款审核列表，支持批量审核。

**工作量估算**

| 任务 | 工作量 |
|------|--------|
| 后端：model/refund.go + controller + 退款条件配置 | 2天 |
| 后端：对接 Epay/微信支付/支付宝退款接口 | 2天 |
| 前端：充值记录退款按钮 + 状态展示 | 1天 |
| 管理后台：退款审核列表 | 1天 |
| **合计** | **6天** |

### 2.4 发票系统（电子发票）

**现状**：无发票功能。

**改造需求**
企业用户充值后通常需要增值税专用发票或普通发票入账。

**方案**

数据模型（新建 `model/invoice.go`）：

```go
type Invoice struct {
    Id          int64   `json:"id" gorm:"primaryKey"`
    UserId      int     `json:"user_id" gorm:"index"`
    TopUpId     int64   `json:"top_up_id"`
    Amount      float64 `json:"amount"`
    InvoiceType int     `json:"invoice_type"` // 1=普通发票 2=增值税专票
    Title       string  `json:"title"`          // 发票抬头
    TaxNumber   string  `json:"tax_number"`     // 税号（专票必填）
    Email       string  `json:"email"`          // 接收邮箱
    Status      int     `json:"status"` // 0=待处理 1=开票中 2=已开具 3=已发送
    InvoiceUrl  string  `json:"invoice_url"`    // PDF 链接
    ApplyTime   int64   `json:"apply_time"`
}
```

开票方式：
- **方案A（推荐，API 自动开票）**：接入百望云 / 航信 / 诺诺发票 API，自动开具并发送至邮箱。
- **方案B（人工开票）**：后台收到申请后，由财务人员手动处理，适合早期规模较小阶段。

方案A单次 API 调用费用约 0.2-0.5 元，适合月发票量 > 100 张的场景。

**工作量估算**

| 方案 | 后端 | 前端 | 合计 |
|------|------|------|------|
| 方案A（API 自动开票） | 3天 | 1天 | **4天** |
| 方案B（人工开票流程） | 1天 | 1天 | **2天** |

**建议**：先上方案B，验证需求量后再接 API 自动化。

代码改动时间：30分钟。**ICP 备案流程本身约 20个工作日**（向云服务商提交，云服务商代为向工信部申请）。

### 3.3 用户协议 + 隐私政策（合规版本）

**现状**
页面框架 `web/src/pages/UserAgreement/` 和 `web/src/pages/PrivacyPolicy/` 已存在，内容通过后端 API 动态配置，无需代码改动。

**改造需求**
现有协议内容若是模板或示例文字，必须替换为符合以下法律的合规版本：
- 《个人信息保护法》（PIPL，2021年11月施行）
- 《网络安全法》（2017年施行）
- 《数据安全法》（2021年施行）
- 《生成式人工智能服务管理暂行办法》（2023年8月施行）

**隐私政策必须覆盖的内容**：
